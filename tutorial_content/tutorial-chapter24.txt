      <Section id="ch24-2" num="24.2" title="Path Sandbox——安全边界">
        <p>所有文件工具共享同一个安全机制：path sandbox。它的职责很简单——拒绝任何试图访问项目目录之外的路径。</p>

        <CodeBlock filename="src/harness/tools/path-sandbox.ts" lang="typescript">{`import { resolve, relative, sep } from "node:path";

export class PathSandbox {
  private readonly allowedBase: string;

  constructor(base?: string) {
    // 默认锚定到进程当前工作目录
    this.allowedBase = resolve(base || process.cwd());
  }

  /** 将用户提供的路径解析为绝对路径。
   *  如果结果在允许范围外则抛出错误。 */
  resolve(userPath: string): string {
    // 从工作目录解析相对路径
    const abs = resolve(this.allowedBase, userPath);

    // 检查是否在 allowedBase 内
    const rel = relative(this.allowedBase, abs);

    // 如果 rel 以 ".." 开头，说明试图逃逸
    if (rel.startsWith(".." + sep) || rel === "..") {
      throw new SandboxEscapeError(userPath, abs, this.allowedBase);
    }

    return abs;
  }

  /** 检查路径是否在允许范围内 */
  contains(testPath: string): boolean {
    const rel = relative(this.allowedBase, resolve(testPath));
    return !rel.startsWith(".." + sep) && rel !== "..";
  }
}

export class SandboxEscapeError extends Error {
  constructor(
    userPath: string,
    resolvedPath: string,
    allowedBase: string,
  ) {
    super(
      \`Path "\${userPath}" resolves to "\${resolvedPath}" \` +
      \`outside allowed base "\${allowedBase}"\`
    );
    this.name = "SandboxEscapeError";
  }
}`}</CodeBlock>

        <Callout kind="warn" title="符号链接的陷阱">
          <p>这个实现有一个已知漏洞：如果项目目录里有指向 <code>/etc</code> 的符号链接，<code>resolve()</code> 会跟着符号链接跑出去。<code>realpath.native()</code> 可以展开符号链接，但性能开销大。产品级实现需要决定：是完全禁用符号链接追踪、在 resolve 时展开、还是在 permission 层额外检查？<br />

        <CodeBlock filename="src/harness/tools/files.ts" lang="typescript">{`import { type Tool } from "./types.js";
import { writeFile, unlink, mkdir } from "node:fs/promises";
import { dirname } from "node:path";
import { PathSandbox } from "./path-sandbox.js";

const sandbox = new PathSandbox();

export const createFileTool: Tool = {
  name: "create_file",
  description: "创建一个新文件。如果父目录不存在会自动创建。",
  inputSchema: {
    type: "object",
    properties: {
      path: {
        type: "string",
        description: "文件路径（相对于项目根目录）",
      },
      content: {
        type: "string",
        description: "文件内容",
      },
    },
    required: ["path", "content"],
  },
  async handler(args: { path: string; content: string }): Promise<string> {
    const safePath = sandbox.resolve(args.path);
    await mkdir(dirname(safePath), { recursive: true });
    await writeFile(safePath, args.content, "utf-8");
    return \`文件已创建: \${args.path}（\${args.content.length} 字符）\`;
  },
};

export const deleteFileTool: Tool = {
  name: "delete_file",
  description: "删除一个文件或空目录。注意：此操作不可恢复。",
  inputSchema: {
    type: "object",
    properties: {
      path: { type: "string", description: "要删除的文件路径" },
    },
    required: ["path"],
  },
  async handler(args: { path: string }): Promise<string> {
    const safePath = sandbox.resolve(args.path);
    await unlink(safePath);
    return \`已删除: \${args.path}\`;
  },
};`}</CodeBlock>

      <Section id="ch24-4" num="24.4" title="list_directory 与 glob_pattern">
        <p>列目录和 glob 是 agent 理解项目结构的两个基本工具。一个精确（list），一个灵活（glob）。</p>

        <CodeBlock filename="src/harness/tools/files.ts" lang="typescript">{`import { readdir, stat } from "node:fs/promises";
import { join, relative } from "node:path";
import { glob } from "glob";

export const listDirectoryTool: Tool = {
  name: "list_directory",
  description: "列出目录中的文件和子目录。返回带类型标记的条目列表。",
  inputSchema: {
    type: "object",
    properties: {
      path: {
        type: "string",
        description: "目录路径（相对于项目根目录，默认根目录）",
      },
    },
  },
  async handler(args: { path?: string }): Promise<string> {
    const safePath = sandbox.resolve(args.path || ".");
    const entries = await readdir(safePath, { withFileTypes: true });

    const lines = entries
      .sort((a, b) => a.name.localeCompare(b.name))
      .map(e => {
        const prefix = e.isDirectory() ? "📁" : e.isSymlink() ? "🔗" : "📄";
        return \`\${prefix} \${e.name}\`;
      });

    return lines.join("\n");
  },
};

export const globPatternTool: Tool = {
  name: "glob_pattern",
  description: "用 glob 模式匹配文件路径。例如 'src/**/*.ts' 匹配所有 src 下的 TypeScript 文件。",
  inputSchema: {
    type: "object",
    properties: {
      pattern: {
        type: "string",
        description: "Glob 模式，支持 **、*、?、{a,b} 等语法",
      },
      path: {
        type: "string",
        description: "搜索根目录（默认项目根目录）",
      },
    },
    required: ["pattern"],
  },
  async handler(args: { pattern: string; path?: string }): Promise<string> {
    const safePath = sandbox.resolve(args.path || ".");

    const results = await glob(args.pattern, {
      cwd: safePath,
      nodir: true,
      dot: true,
      ignore: ["**/node_modules/**", "**/.git/**", "**/dist/**"],
    });

    if (results.length === 0) {
      return "没有匹配的文件。";
    }

    // 限制输出大小——防止 2000 个匹配塞爆 context
    const maxResults = 100;
    const lines = results.slice(0, maxResults);
    const summary = results.length > maxResults
      ? \`\n…以及另外 \${results.length - maxResults} 个匹配文件\`
      : "";

    return lines.join("\n") + summary;
  },
};`}</CodeBlock>

        <Callout kind="tip" title="glob vs 模型直接做文件发现">
          <p>一个有趣的现象：给模型 <code>list_directory</code> 它会一步步探索，像人类一样"看这个目录→进去→再看"。这需要多回合。给 <code>glob_pattern</code> 它能一步到位。关键是模型需要知道用哪个模式。<br />
          最佳实践：先让模型试 glob，如果没找到再用 list_directory 确认。</p>
        </Callout>

      <Section id="ch24-5" num="24.5" title="find_text_in_files 与 get_file_info">
        <p>文本搜索和文件元信息：agent 版的 grep 和 stat。</p>

        <CodeBlock filename="src/harness/tools/files.ts" lang="typescript">{`import { stat } from "node:fs/promises";

export const findTextInFilesTool: Tool = {
  name: "find_text_in_files",
  description: "在文件中搜索文本（类似 grep）。返回匹配的文件路径和行号。",
  inputSchema: {
    type: "object",
    properties: {
      pattern: {
        type: "string",
        description: "搜索文本（支持子串匹配，非正则）",
      },
      glob: {
        type: "string",
        description: "限定文件模式，例如 '*.ts' 或 'src/**'",
      },
      maxResults: {
        type: "number",
        description: "最大结果数（默认 30，避免太多结果）",
      },
    },
    required: ["pattern"],
  },
  async handler(
    args: { pattern: string; glob?: string; maxResults?: number },
  ): Promise<string> {
    const maxResults = Math.min(args.maxResults ?? 30, 200);
    const root = sandbox.resolve(".");
    const results = await grepInDir(root, args.pattern, {
      glob: args.glob,
      maxResults,
      ignore: ["node_modules", ".git", "dist"],
    });

    if (results.length === 0) {
      return "没有找到匹配的内容。";
    }

    return results
      .map(r => \`\${r.file}:\${r.line}: \${r.text.trim()}\`)
      .join("\n");
  },
};

export const getFileInfoTool: Tool = {
  name: "get_file_info",
  description: "获取文件的元信息——大小、修改时间、类型。",
  inputSchema: {
    type: "object",
    properties: {
      path: {
        type: "string",
        description: "文件或目录路径",
      },
    },
    required: ["path"],
  },
  async handler(args: { path: string }): Promise<string> {
    const safePath = sandbox.resolve(args.path);
    const info = await stat(safePath);

    const lines = [
      \`路径: \${args.path}\`,
      \`类型: \${info.isDirectory() ? "目录" : info.isFile() ? "文件" : "其他"}\`,
      \`大小: \${formatSize(info.size)}\`,
      \`修改时间: \${info.mtime.toISOString()}\`,
    ];

    if (info.isFile()) {
      // 对于小文件，顺便提示代码预览
      if (info.size < 100_000) {
        lines.push("提示: 小文件，可直接用 read_file 查看");
      } else {
        lines.push(\`提示: 文件较大 (\${formatSize(info.size)})，建议用 read_file viewport 分段查看\`);
      }
    }

    return lines.join("\n");
  },
};

function formatSize(bytes: number): string {
  if (bytes < 1024) return \`\${bytes} B\`;
  if (bytes < 1024 * 1024) return \`\${(bytes / 1024).toFixed(1)} KB\`;
  return \`\${(bytes / (1024 * 1024)).toFixed(1)} MB\`;
`}</CodeBlock>

        <Callout kind="key" title="搜索工具与模型能力的交互">
          <p><code>find_text_in_files</code> 返回格式是 <code>file:line: text</code> —— 和 grep 一样。这不是巧合：模型训练数据里有大量这种格式。选择模型熟悉的格式可以减少 parse 错误。<br />
          同理 <code>get_file_info</code> 返回的是 key:value 对——简单、稳定、模型容易从中提取信息。</p>
        </Callout>
      </Section>

        <CodeBlock filename="src/cli/permissions.ts" lang="typescript">{`// 在 cliPermissionHandler 中补充文件操作策略

const FILE_RISK_MAP: Record<string, "none" | "low" | "medium" | "high"> = {
  read_file: "none",
  read_file_viewport: "none",
  list_directory: "none",
  get_file_info: "none",
  glob_pattern: "none",
  find_text_in_files: "low",
  create_file: "medium",
  edit_lines: "medium",
  delete_file: "high",
};

export async function cliPermissionHandler(
  req: PermissionRequest,
): Promise<PermissionDecision> {
  const risk = FILE_RISK_MAP[req.action] ?? "medium";

  if (risk === "none") {
    return { allowed: true, reason: "read-only" };
  }

  if (risk === "low") {
    console.log(\`🔍 \${req.action} (\${req.details})\`);
    return { allowed: true };
  }

  // medium / high —> 需要交互式确认
  return await interactiveConfirm(req, risk);
}`}</CodeBlock>
