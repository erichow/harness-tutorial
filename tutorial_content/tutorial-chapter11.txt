/* ─── Four ACI principles cards ─── */
function FourACIPrinciples() {
  return (
    <div className="cards cols-2">
      <Card title="① Viewport, not dump" kind="">
        模型读 2000 行文件用一个工具调用时，<strong>它对那 2000 行没有任何结构性能力</strong>——不能 scroll、不能视觉搜索、不能持有"我在哪"的心智地图。
        <br /><br />更好：一个返回<strong>窗口（50-100 行）</strong>的工具，带显式位置指示器和"滚动"命令。
      </Card>
      <Card title="② Targeted edit, not rewrite" kind="">
        想改 2000 行文件的第 47 行，<strong>不该返回整 2000 行</strong>。该返回"变化"。
        <br /><br />Targeted edit 还让<em>意图可审计</em>——diff 最小、review 容易、revert 简单。
      </Card>
      <Card title="③ Explicit envelopes" kind="">
        每个工具返回都需要一个机器可读的<strong>外壳</strong>：返回了什么、被截断了什么、下一步该怎么做。
        <br /><br /><code style={{ fontSize: 12 }}>[file: /etc/passwd; lines 1-100 of 423; call again with offset=100 for more]</code> — 写起来便宜，<em>省去模型"猜"</em>。
      </Card>
      <Card title="④ Errors as instructions" kind="">
        <em>"File not found"</em> 是信息。
        <br /><strong>"File not found: /etc/passwd. Did you mean /etc/passwd.bak (fuzzy match)? Or use list_files('/etc')"</strong> 是指令。
        <br /><br />模型对<strong>指令</strong>反应更好。
      </Card>
    </div>
  );
}

/* ─── Interactive viewport demo ─── */
function ViewportInteractiveDemo() {
  const [offset, setOffset] = useS11(0);
  const total = 423;
  const limit = 100;
  const end = Math.min(total, offset + limit);
  // synthetic content
  const lines = Array.from({ length: limit }, (_, i) => {
    const n = offset + i + 1;
    if (n > total) return null;
    const stub = [
      "import asyncio", "from dataclasses import dataclass",
      "from typing import Callable", "", "@dataclass",
      "class Tool:", "    name: str", "    description: str",
      "", "    def run(self, **kwargs) -> str:",
      "        return self._fn(**kwargs)",
    ];

  }).filter(Boolean);

from pathlib import Path

from .base import Tool
from .decorator import tool


VIEWPORT_DEFAULT = 100
VIEWPORT_MAX = 500


@tool(side_effects={"read"})
def read_file_viewport(
    path: str, offset: int = 0, limit: int = VIEWPORT_DEFAULT
) -> str:
    """Read a slice of a text file, like \`less\` or \`head -n ... | tail -n ...\`.

    path: filesystem path.
    offset: zero-based line number to start reading from. Default 0.
    limit: max lines to return. Default 100, max 500.

    Returns a rendered viewport with line numbers. The last line of the
    output describes what's visible and what's NOT, so you can call this
    tool again with a different offset to keep reading.

    Side effects: reads the filesystem.

    Use this in preference to reading whole files. For files <50 lines,
    the whole file fits in one call.
    """
    limit = min(max(1, limit), VIEWPORT_MAX)
    p = Path(path)
    if not p.exists():
        raise FileNotFoundError(f"file does not exist: {path}")
    if not p.is_file():
        raise IsADirectoryError(f"not a regular file: {path}")

    lines = p.read_text(encoding="utf-8", errors="replace").splitlines()
    total = len(lines)
    start = max(0, offset)
    end = min(total, start + limit)
    visible = lines[start:end]

    width = len(str(total))
    numbered = [
        f"{i + 1:>{width}}  {line}"
        for i, line in enumerate(visible, start=start)
    ]
    body = "\\n".join(numbered)
    footer = (
        f"\\n[file {path}; lines {start + 1}-{end} of {total}"
        + (f"; MORE below — call with offset={end}" if end < total else "; end of file")
        + (f"; MORE above — call with offset=0" if start > 0 else "")
        + "]"
    )
    return body + footer`}</CodeBlock>

      <Section id="ch11-3" num="11.3" title="行范围编辑器">
        <CodeBlock filename="src/harness/tools/files.py（续）" lang="python">{`@tool(side_effects={"write"})
def edit_lines(
    path: str,
    start_line: int,
    end_line: int,
    replacement: str,
) -> str:
    """Replace a line range in a file with new content.

    path: filesystem path (file must exist).
    start_line: one-based starting line (inclusive).
    end_line: one-based ending line (inclusive).
    replacement: text to insert in place of the removed lines. Empty
                 string deletes the range without replacement. Include
                 trailing newlines if you want blank lines.

    Returns a confirmation with the diff summary and the lines around
    the edit (for verification).

    Side effects: writes the file. Preserves content outside the range.

    To INSERT new lines at position N without removing: use
    start_line=N, end_line=N-1 and replacement=your_new_content.
    To APPEND: use start_line=last+1, end_line=last.
    """
    p = Path(path)
    if not p.exists():
        raise FileNotFoundError(f"file does not exist: {path}")

    original = p.read_text(encoding="utf-8")
    lines = original.splitlines(keepends=True)
    total = len(lines)

    if start_line < 1 or start_line > total + 1:
        raise ValueError(
            f"start_line {start_line} out of range (1..{total + 1})"
        )
    if end_line < start_line - 1 or end_line > total:
        raise ValueError(
            f"end_line {end_line} out of range ({start_line - 1}..{total})"
        )

    # Normalize: start is zero-based slice, end is zero-based exclusive
    s = start_line - 1
    e = end_line   # slice end is exclusive of end_line; works for deletes too

    replacement_lines = replacement.splitlines(keepends=True)
    if replacement and not replacement.endswith("\\n"):
        # Don't glue onto the next line without a newline
        if e < total:
            replacement_lines[-1] = replacement_lines[-1] + "\\n"

    new_lines = lines[:s] + replacement_lines + lines[e:]
    p.write_text("".join(new_lines), encoding="utf-8")

    removed = end_line - start_line + 1 if end_line >= start_line else 0
    added = len(replacement_lines)

    # Render context around the edit
    context_start = max(0, s - 2)
    context_end = min(len(new_lines), s + len(replacement_lines) + 2)
    preview = "".join(
        f"{i + 1:>5}  {new_lines[i]}"
        for i in range(context_start, context_end)
    )
    return (
        f"edited {path}: removed {removed} lines, "
        f"added {added} lines at {start_line}..{end_line}\\n"
        f"context:\\n{preview}"
    )`}</CodeBlock>

      <Section id="ch11-5" num="11.5" title="把 Envelope 模式扩展到 bash">
        <p>Viewport 模式专属于文件，但<strong>显式 envelope 原则可推广</strong>。每个可能大的工具输出都该有同样的形状：</p>

        <CodeBlock filename="envelope shape" lang="text">{`<content>
[tool_result: <N> items/lines/bytes returned; <M> more omitted.
 Call <suggestion> to see more.]`}</CodeBlock>

        <CodeBlock filename="src/harness/tools/std.py（bash 更新）" lang="python">{`BASH_OUTPUT_LIMIT = 4000   # characters


@tool(side_effects={"read", "network"})
def bash(command: str, timeout_seconds: int = 30) -> str:
    """Run a shell command in the current working directory.
    [... description ...]
    """
    import subprocess
    timeout = min(int(timeout_seconds), 300)
    result = subprocess.run(
        command, shell=True, capture_output=True, text=True, timeout=timeout,
    )
    out = result.stdout
    err = result.stderr

    out_truncated = len(out) > BASH_OUTPUT_LIMIT
    err_truncated = len(err) > BASH_OUTPUT_LIMIT // 2

    if out_truncated:
        out = out[:BASH_OUTPUT_LIMIT] + f"\\n...[truncated at {BASH_OUTPUT_LIMIT} chars]"
    if err_truncated:
        err = err[:BASH_OUTPUT_LIMIT // 2] + f"\\n...[truncated]"

    note = ""
    if out_truncated or err_truncated:
        note = (
            "\\n[note: output was truncated. For large output, pipe through "
            "\`head\`, \`tail\`, \`grep\`, or save to a file and use read_file_viewport.]"
        )

    return (
        f"exit={result.returncode}\\n"
        f"---stdout---\\n{out}\\n"
        f"---stderr---\\n{result.stderr}"
        + note
    )`}</CodeBlock>

        <Callout kind="key">
          bash 现在 cap 输出、显式标注截断、<strong>告诉模型怎么处理</strong>。建议（"pipe through head/tail/grep"）是 LLM-aware 的小设计——这是<em>缩减输出的惯用 shell 方式，模型已经认识这些工具</em>。
        </Callout>

        <p>把同样思路应用到任何会返回多的工具：第 10 章的 <code>search_docs</code>、<code>scratchpad_read</code> 在 entry 巨大时、你加的任何 HTTP GET 工具。<strong>跨工具的 envelope 一致性本身就是 feature</strong>——模型学一次形状，到处适用。</p>
      </Section>

      <Section id="ch11-6" num="11.6" title="描述卫生（Description hygiene）">

            <br /><br />模型知道：副作用、异步语义、幂等键、行为约束。<strong>难误用</strong>。

        <ol style={{ fontFamily: "var(--font-hand)", fontSize: "1.05em", lineHeight: 1.65 }}>
          <li><strong>What it does</strong>——一句话</li>
          <li><strong>What it requires</strong>——前置条件：文件存在、user 存在、process 在跑</li>
          <li><strong>What it does NOT do</strong>——scope 限制："不抓 URL"、"不改 git state"</li>
          <li><strong>Side effects</strong>——read / write / network / mutate，明确写出</li>
          <li><strong>Output envelope</strong>——返回值长什么样，包括截断行为</li>
          <li><strong>When to prefer it</strong>——"用这个而非 X 当..."</li>
        </ol>

        <p>原版 SWE-agent ACI 包含一堆<em>定制命令</em>：<code>find_file</code>、<code>search_dir</code>、<code>create</code> 和一个细致的文件 viewer 状态机。<strong>Mini-SWE-agent 跟进版几乎全砍了，只用 bash——SWE-bench 成绩相近，~100 行代码</strong>。</p>

        <p>变了什么？<strong>前沿模型在通用工具上的使用能力变好了</strong>。SWE-agent 为补偿 GPT-4 的笨拙建的那套精致 ACI 命令，到了 Claude 3.5 之后<em>不必要了</em>——只要 output 框架得好，它们能熟练驱动一个 shell。</p>

        <Callout kind="key" title="教训">
          <p><strong>设计工具来弥补模型的弱点，不要去重造它已有的能力。</strong></p>
          <ul>
            <li>Viewport 读仍然值得——任何模型对 5 万 token 的工具输出都不行</li>
            <li>行范围编辑仍然值得——它是 diff 的本质，让 agent 意图可审计</li>
            <li><strong>但</strong>当模型能调 bash 时，重新实现 <code>ls</code> 或 <code>grep</code> 很少值得这个维护负担</li>
          </ul>
          <p>本书设计正好在这个甜点上：<em>加约束 token 流的（viewport、envelope），让模型用 bash 处理通用情况</em>。</p>
        </Callout>
      </Section>

      <Section id="ch11-8" num="11.8" title="试一试">
        <div className="cards cols-3">
          <Card title="① 测量 token 影响">
            跑一个读 1000 行文件的任务：先用旧 <code>read_file</code>，再用 <code>read_file_viewport</code>。对比总 token 消耗 + 输出质量。<strong>viewport 永远更好，还是只对大文件？</strong>
          </Card>
          <Card title="② 给 edit 加 dry_run">
            加一个 <code>dry_run=True</code> 参数，返回 diff 但不写。Agent 能用它在 commit 前验证。<strong>dry-run 引入哪些权衡？</strong>
          </Card>
          <Card title="③ 故意写一个坏工具">
            写一个 <code>read_file_unbounded</code>——返回整个文件、没有 envelope。把它和 viewport 版一起给 agent。看模型挑哪个。<strong>prompt 短的时候它会偏向更差的那个吗？</strong>这告诉你 description discipline 的价值。
          </Card>
        </div>
      </Section>
